- **切换对话/离开页面的流式竞态设计正确**(页面不缓存,新实例隔离;跨会话恢复气泡逻辑自洽)。
- **分支模型持久化**(`SyncActiveBranch` 回填机制)本身不丢文本数据——丢的只是 P0-5 列出的未复制字段。
- `OobeCompleted` 置位时机、设置即改即存 + `_loading` 门控、删除操作的二次确认(除 P0-11/12 清理不彻底)、`TryParseHex` 防护、工具循环的取消/退避/密钥防泄漏、`LastCompactAt` 用 Ticks 绕序列化坑——均正确。

## P3 —— 功能缺口(开发者自报,已代码核实)

这些不是 bug,是设计层面的缺口。按现状 → 建议列出:

### P3-1 工具调用过程不持久化(已核实)
`ConversationMessage`(`Models/Models.cs`)**没有任何 ToolCalls/工具轨迹字段**。工具循环(`Tools.cs RunFunctionCallLoopAsync`)的中间 assistant(tool_calls)/tool(result) 消息只存在于循环内的内存 List,循环结束后仅最终文本进 `conv.Messages`。后果:
- 重启后工具调用轨迹全部消失,气泡里"调用了什么工具、参数、结果"不可回看;
- 多轮对话重建请求历史时不含既往工具轮次,模型失去工具上下文;
- 与 P1-4 同根:`ReasoningContent` 在工具轮次间也被丢弃。

**建议**:给 `ConversationMessage` 加 `List<ToolCallRecord>`(工具名/参数摘要/结果摘要/耗时/状态,`[DataMember]`),循环每轮落一条;结果按上限截断存储防存档膨胀;气泡内折叠渲染"🔧 调用了 N 个工具(展开)"。展示层先做,是否回传给 API 可以后议(回传需处理 P1-6 的顺序问题)。

### P3-2 思考内容在工具轮次中丢失(与 P1-4 同根)
`ReasoningContent`/`ThinkSteps` 字段本身是 `[DataMember]`、能持久化(Models.cs:141-144),**但只有最终消息的思考被保存**;工具循环中每一轮的思考在重建 assistant 消息时被丢弃(`Tools.cs:1042-1048`),且 `SerializeMessage` 不序列化它。修 P1-4 时顺带把每轮思考累积进最终消息的 `ThinkSteps`,一次解决两个问题。

### P3-3 ask_user 面板:单字段 TCS,存在并发覆盖 bug(已核实,升级为 bug)
`MainPage.Subagents.cs:80-122`:`_askTcs`/`_askViewModels` 是**页面级单字段**,且 `AskUserPanelCallback` 与 `ShowPermissionPanel`(:125)共用。而工具是 `Task.WhenAll` 并行执行的(`Tools.cs:1086`)——两个工具同时 ask_user(或 ask_user 与权限确认同时弹)时,第二个调用直接覆盖第一个的 TCS,**第一个 await 永远不完成 → 工具循环挂死**。另外:
- 无超时:用户导航离开页面,TCS 永不完成,循环永挂;
- 取消返回 null,模型侧收到什么取决于调用方,语义模糊;
- 面板是页面级控件,不支持子代理内的提问。

**建议**:TCS 改为局部变量 + 请求队列串行弹出;加超时(如 120s 返回"用户未响应");页面 `OnNavigatedFrom` 时对未决 TCS 置取消结果。

### P3-4 子代理体验(与 H5/P2-20/L5 同根)
深度上限是全局静态(A 对话占名额 B 无法 spawn)、Tracker 跨线程读写状态无同步、子代理无独立轨迹持久化。**建议**:深度计数挂到对话/调用链上;子代理的完整轨迹(含它自己的工具调用)存成可展开的记录,而不是只有一行状态。

### P3-5 无 workflow / loop(定时、循环、多步编排)
当前完全没有。不建议直接上完整 workflow 引擎,MVP 路线:
1. **循环任务**:一个后台 runner,按间隔重发指定 prompt 到指定会话,带停止条件(最大次数/用户停止),UI 上一个"循环任务"管理列表——复用现有发送管线即可;
2. **任务链**:顺序执行的 prompt 列表,上一步输出可注入下一步(占位符),失败即停;
3. 以上都稳定后再考虑图状/并行编排。
注意先修 P1-24(单 `_streamCts` 停不了多任务)再做这个,否则循环任务一多停止按钮必乱。
